# CLAUDE.md の書き方

CLAUDE.md を更新する際は、必ずこのドキュメントの基準に従うこと。

## 基本原則

CLAUDE.md は毎セッション必ずコンテキストに読み込まれる。短いほど1行あたりの重みが増し、長いほど本当に守ってほしい記述が埋もれる。

書く基準は次の2つだけ。

- **コードから導出できない事実**（コードを全部読んでも分からないこと）
- **世間のデフォルトと違う判断**（標準的なやり方が複数ある場面で、このプロジェクトが選んだ側）

古くなった記述はゼロより悪い（モデルが古い情報を信じて間違える）。増やすより削ることを優先する。

## 書く場所の振り分け

| 内容 | 置き場所 |
|---|---|
| 特定のファイルパス・ファイル種別に紐づく制約・規約 | `.claude/rules/*.md`（globスコープ） |
| 手順・やり方 | Agent Skills |
| 深い参照知識（用語集・概要・環境情報など） | `docs/` に置き、CLAUDE.md からリンク |
| プロジェクト全体に効く制約・流儀 | CLAUDE.md |

見分け方: **特定の種類のファイルを編集しているときだけ必要**なら rules、**設計・計画・レビューなど何をするときでも前提**になるなら CLAUDE.md。

## 「制約」セクションに書くこと

見た目やコードからは危険と判断できない、プロジェクト全体に効く禁止・必須事項のみ。

- 理由を必ず添える。理由があれば、書いていないケースにもモデルが類推を効かせられる
- 一般的な注意（テストを書く、秘密情報をコミットしない等）はモデルが知っているので書かない

例: 「migrationを勝手に実行しない（理由: 開発DBはチーム共有のため）」

## 流儀（必要になったら見出しを足す）

標準的なやり方が複数ある場面で、このプロジェクトが選んだ側とその理由。最初は無くてよい。モデルがデフォルトと違うことをやって直させた瞬間に、1個ずつ書き足していく。

例:

- 「日時は内部では全部UTC、表示層でのみJSTに変換」
- 「エラーはResult型で返す。例外は使わない（理由: 呼び出し側にハンドリングを強制するため）」

書くべきでないもの:

- 世間のデフォルト通りの選択（モデルが勝手にそうする）
- 一般的なベストプラクティス（モデルが知っている）
- ディレクトリ構成やレイヤー設計の説明（コードを見ればわかる）
