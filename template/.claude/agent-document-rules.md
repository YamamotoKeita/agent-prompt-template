ローカルファイルで仕様などを管理する。
機能・作業の単位ごとに `.work/<機能・作業名>/` を作り、その中を **ssot（確定）** と **ai-work（AI出力）** に分ける。

- .work/<機能・作業名>/
  - ssot <- 要望や要求、動作仕様。人間が書くか人間が確認して承認したものだけ置く。AIは承認のないものをここに出力しない。
    - 要件・要望.md - 見出しで「背景・課題・望み（なぜ必要か）」「要望（原文・改変禁止）」「要件」を分ける
    - 詳細仕様.md - 承認済みの動作レベル仕様
    - 意思決定.md - 方針のみ（"何を採用＋なぜ"まで／"どう実装"は書かない）。ADR的でその時点の記録・変更あり得る（古いものは supersede として残す）
    - 知見.md - 人間が検証済みの学び
  - ai-work <- AI出力。"真実"ではない／AIが決めたものは絶対ここから出さない
    - 詳細設計.md - 承認されても ssot には上げない
    - 自己判断決定ログ.md - AIの自己判断の記録
    - その他必要に応じて作成（調査メモ・未検証の知見 など）

運用ルール：

- **AIはまず `ai-work` にドラフトを出す。** 必須確認①/②/③で人間が承認したものだけ `ssot` へ昇格する。
- **作業の根拠にしてよいのは `ssot` だけ。** `ai-work` は未確定であり、真実として依拠しない。
- **詳細設計・AIが決めたもの・未検証の知見は `ssot` に入れない。** 知見は人間が検証して初めて `ssot/知見.md` へ昇格する。
- 完了したら `work/<機能・作業名>/` の ssot をプロジェクト全体の `docs/` へ昇格 or アーカイブする（MVPでは任意）。
