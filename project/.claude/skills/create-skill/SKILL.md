name: create-skill
description: スキル (Agent Skill)を作成・更新する。再利用可能な作業手順などをスキルとして定義する際利用する。SKILL.md と必要に応じて付随する scripts や references を作成する。

## Agent Skillのファイル構成
```
{プロンプト配置ディレクトリ}/skills/{スキル名}/
├── SKILL.md          # 必須: メイン指示
├── scripts/          # 任意: 実行可能コード
├── references/       # 任意: 必要に応じて読む詳細ドキュメント
└── assets/           # 任意: 出力用素材（コンテキストに読み込まない）
```

1スキル = 1能力で定義する。トリガーワード・入出力が別物なら分割する。

SKILL.mdは以下のフォーマットにする。

```yaml
---
name: your-skill-name
description: スキルの説明。何をするか、いつ使うか、トリガーワードなど。
---
スキルの本文。手順やルールなど。
```

## Front matterのルール
Front matterには `name` と `description` を記載する。

### `name`
- 小文字・数字・ハイフンのみ
- 64文字以内
- XMLタグを書かない
- 予約語（`anthropic`, `claude`）を使わない

### `description`
何をするか、いつ使うかを記述する。Skill発火のトリガーとなる。
競合を避けるため、なるべく具体的な用語を含め、トリガーにしたい同義語がある場合はそれらも含める。
基本の構成は `何をする + いつ使う + 補足やトリガーにしたい同義語` 。
`何をする`の部分は主語を書かず動詞の基本形で終える。

以下のルールに従う。

- 空にしない
- 最大1024文字
- XMLタグを書かない

#### descriptionの例
- ❌ `description: ドキュメントを扱う`
- ✅ `description: Excelスプレッドシートを分析し、ピボットテーブルを作成する。Excelファイル、.xlsxファイルを分析する際に使用。`

## SKILL.md 本文
AI エージェントは既に賢いため一般的な知識を持つ前提で、必要最小限の手順・情報・制約・例などを記載する。

**500行以内**に収める。超えそうなら詳細を `references/` に分離。
**記述トーン**: 命令形/不定詞で統一する（「〜する」「〜を確認」）。説明文体（「〜します」「〜できます」）は使わない。

### 指示の厳密さ
指示の具体度は内容やユースケースに応じて調整する。
タスクの壊れやすさと許容されるばらつきに応じて、指示の自由度を選ぶ。

- 高自由度: 複数のやり方が妥当で、状況判断が重要なタスクに使う。短い方針や観点だけを書く。
- 中自由度: 推奨パターンはあるが多少の調整を許したいタスクに使う。擬似コード・テンプレート・代表例を置く。
- 低自由度: 手順を誤ると壊れやすいタスクに使う。実行コマンド、固定順序、禁止事項、検証手順を明示する。

迷った場合は、高自由度から書き始めず、失敗コストに見合う厳密さまで具体化する。

### 指示のパターン
固定のフォーマットはないがいくつか定番のパターンがあるため以下に説明する。

#### テンプレートパターン
スキルの出力を厳密に指定したい場合は出力のテンプレートを記載する。APIレスポンスなどフォーマットが固定なデータを扱う場合に有効。柔軟性も与えたい場合は、テンプレートを一例として、必要に応じた最適化を許可する旨記載する。

#### 入出力例パターン
説明より例が効果的な場合、スキルへの入力とそれに対して望まれる出力のペアを例示する。
例は具体的で多様なケースをカバーするようにし、必要に応じて説明を加える。

例:
````md
以下の例に従ってコミットメッセージを生成する。

**例1**
入力: JWT を使ったユーザー認証を追加した
出力:
```text
feat(auth): JWT ベースの認証を実装
ログイン API とトークン検証処理を追加
```

**例2**
入力: レポートの日付表示の不具合を修正した
出力:
```text
fix(reports): タイムゾーン変換時の日付表示を修正
レポート生成で UTC の扱いを統一
```

**例3**
...
````

#### 条件分岐ワークフローパターン
判断ポイントがある作業では、分岐条件を明示し、条件ごとに進む手順を分けて記載する。
分岐先の手順が長い場合は SKILL.md には判断基準だけを書き、詳細は対応する別ファイルへ直接リンクする。

例:
````md
ドキュメント更新フロー
1. 作業種別を判定する
	**新規作成** → 「新規作成フロー」を実行
	**既存編集** → 「既存編集フロー」を実行

2. 新規作成フロー
	- テンプレートを使って下書きを作成
	- 必須項目を埋める
	- 出力形式を確認する

3. 既存編集フロー
	- 対象ファイルを読む
	- 変更箇所だけ更新する
	- 影響範囲を確認する
````

#### 複雑なタスクのワークフローパターン
複雑な作業は、飛ばしてはいけない確認や検証を含めて、順番のある手順に分解して記載する。
工程が多い場合は、応答にそのまま写して進捗管理できるチェックリストを添える。
各ステップでは「何をするか」だけでなく、必要なら「失敗したらどこに戻るか」まで明示する。

例:
````md
このチェックリストをコピーして進捗を管理する。

```
調査進捗:
- [ ] Step 1: 関連資料をすべて読む
- [ ] Step 2: 主要な論点を整理する
- [ ] Step 3: 根拠を突き合わせる
- [ ] Step 4: 要約を作成する
- [ ] Step 5: 引用元を確認する
```

1. 関連資料をすべて読む
2. 主要な論点を整理する
3. 根拠を突き合わせる
4. 要約を作成する
5. 引用漏れがあれば Step 3 に戻る
````

## Progressive disclosure

SKILL.md が肥大化する場合、詳細や参考情報を `references/` に分離する。

- 参照ファイルは **SKILL.md から直接リンク**する（ネスト参照禁止）
- 100行超の references は冒頭に目次を付ける
- サイズが大きい（>10k words）references は、SKILL.md に grep パターンを記載して部分参照を可能にする
- 同じ情報を SKILL.md と references の両方に書かない

## scriptsの利用
決定的で繰り返し使う処理はscript化し、`scripts/` に配置する。
以下のルールに従う。

- 必要な依存関係と実行前提を SKILL.md に明記する
- `scripts/` には実行するscriptのみ配置する。参考情報としての script は `references/` に配置する。
- 破壊的変更や複雑な処理では、計画生成 → 検証 → 実行 → 結果確認のループを入れる
- script のエラー出力は、何を直せば次に進めるか分かる形にする

## PATHの記述ルール
SKILL.md 内にscripts や references へのパスを書く場合は、スキルルートを基準にした相対パスで記載する。
例: `scripts/hoge.sh`

## その他の注意点

- time-sensitive な情報を本文に書かない（現在推奨のみ記載）
- MCP ツール参照時は完全修飾名を使う（`ServerName:tool_name`）
- Pathの区切り文字には/を使う（\は使わない）

## Checklist

スキル作成・更新後の最終確認:
- [ ] `name` と `description` が要件を満たす
- [ ] 説明が十分に簡潔で過剰でない（AI エージェントは既に一般的なことは知っている前提）
- [ ] SKILL.md 本文が500行未満
