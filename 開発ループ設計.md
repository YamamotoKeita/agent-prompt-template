# AI主導 開発ループ 設計

## 狙い
ジュニアエンジニアが「これ作って」とメインのClaude Codeに言うだけで、要件詰め→計画→実装→検証までを **AIが主導して自走** で回す。人間は **要所2ゲートだけ** 確認する。

## 役割分担（ここが肝）

| 道具 | 役割 | 人間と対話 | 使いどころ |
|---|---|---|---|
| **orchestrator Skill** | 司令塔。ループ全体の進行を駆動する | ◎ する | メインループに常駐。人間の窓口 |
| **Skill（各工程）** | interview / planning / build / git など | ◎ する | 各工程の実作業手順 |
| **Subagent** | 作業員。閉じたタスクを委任 | ✗ 投げっぱなし | 実装・調査など重い単発作業 |
| **Workflow** | 裏で並列ファンアウト | ✗ 途中で人に聞けない | 多観点の検証・多重レビュー |

> ⚠️ 司令塔は **Subagentにしない**。人間が話しかける相手はメインループであって、Subagentは結果を返して消える使い捨てだから。
> ⚠️ ループ全体を **Workflowで回さない**。Workflowは裏で走るので途中の人間確認ができない。並列したい一区間だけで呼ぶ。

## ループ全体図

```
人間（ジュニア）── 要望 ──▶ メインループ ＝ orchestrator Skill が駆動
   │
   ├─ ① interview skill         要件・エッジケース・曖昧さを詰める
   ├─ ②(デザイン作成)            必要時のみ
   ├─ ③ planning skill          実装計画 ＋ テスト計画
   │
   ├─ ▶▶ ゲート①：計画を人間に確認 ◀◀  ← ここで止まる
   │
   ├─ ④ 実装                     subagentに委任（git worktreeで隔離）
   │      └─ 仕様の重大な穴を見つけたら割り込み質問（例外的）
   ├─ ⑤ ユニットテスト           build / run-test skill
   ├─ ⑥ 検証（Workflowで並列4観点）
   │      ├─ テストをパスするか
   │      ├─ 仕様を満たすか
   │      ├─ 要件を満たすか
   │      └─ クライアントの真なる望みを叶えるか
   │
   └─ ▶▶ ゲート②：検証結果を人間に提示 ◀◀  ← ここで止まる
```

## 2つのゲート（要所だけ止まる）

- **ゲート①（計画確定）**: 実装に入る前。計画とテスト計画を提示し、承認/修正をもらう。手戻りコストが一番高い場所なので必ず止める。
- **ゲート②（最終検証）**: 検証結果サマリと差分を提示。ここでOKなら次のステップ（PR等）へ。

それ以外は **AIが自走**。ただし「仕様の前提が崩れる重大な発見」をした時だけ、ゲート外でも人間に割り込み質問してよい（暴走防止の安全弁）。

## 検証フェーズだけWorkflowを使う理由

検証の4観点は互いに独立で、並列に走らせて最後にまとめるのが効率的。`Workflow` のpipeline/parallelで4観点を並列検証 → 各指摘を別agentで反証検証（adversarial verify）→ 確定したものだけゲート②で提示。
※ Workflowは人に聞けないので、**観点定義・合否基準はゲート①の計画段階で確定**しておくのが前提。

## 既存Skillのマッピング

| 工程 | 使うSkill（既存/新規） |
|---|---|
| 要件詰め | `interview`（既存） |
| 計画 | `planning-implementation` / `planning-overview-design`（既存） |
| 実装 | subagent委任 ＋ `git`(worktree) |
| ビルド/テスト | `build`（既存）/ `run-test`（新規） |
| 検証 | `verify`（既存）＋ Workflow並列 |
| レビュー | `code-review` / `code-review-carefully`（既存） |
| コミット | `git` / `github-make-pull-request`（既存） |
